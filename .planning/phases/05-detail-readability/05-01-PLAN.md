# Phase 5 Plan 01: Rewrite format_pi_text

**Phase:** 05-detail-readability
**Plan:** 01
**Subsystem:** build-pipeline
**Depends on:** Phase 3 (build.py, format_pi_text, drug_detail.html)

## Goal

Fix the `format_pi_text` function so that FDA prescribing information renders as readable, scannable HTML with proper sub-section headings, bullet lists, and paragraph breaks — instead of walls of run-on text.

## Root Cause Analysis

The current `format_pi_text` (build.py:41-166) has three bugs:

### Bug 1: Split regex triggers on cross-reference patterns

The split regex `(?=(?:^|\s)(\d+(?:\.\d+)+\s+[A-Z]))` (line 75) matches ANY occurrence of `N.N <capital letter>`, including:
- In parenthesized cross-refs: `"( 1.1 ) 1.1 Plaque Psoriasis"` — the `1.1 P` inside `( 1.1 )` triggers the split, producing a phantom part `"1.1 P"` (the "P" from "Plaque")
- This phantom part has title="P" which fails the `len(m.group(2)) > 2` check
- But the split has already occurred, so the *real* heading "1.1 Plaque Psoriasis" becomes a separate part that also fails (it starts with "1.1 " again but by then the second split has already consumed it differently)

**Impact:** Cosentyx indications renders as a single `<p>` with 0 `<h4>` subsections — all 6 indications are lost.

### Bug 2: Indication lead-in text isn't parsed into lists

The typical FDA pattern is:
```
DRUG is indicated for the treatment of:
  moderate to severe plaque psoriasis (PsO) in adults... ( 1.1 )
  active psoriatic arthritis (PsA) in adults... ( 1.2 )
  adults with active ankylosing spondylitis (AS). ( 1.3 )
```

Currently this renders as a single `<p>` with inline `<span class="xref">` markers. The cross-refs mark natural list boundaries that should be `<li>` items.

**Impact:** 404/530 drugs (76%) have indication lead-in patterns. Currently renders as an unreadable run-on sentence.

### Bug 3: No paragraph splitting for long text

4,362 `<p>` blocks exceed 3,000 chars. The longest is 36,866 chars (Venclexta). There is no logic to break long text at sentence boundaries.

**Impact:** 84.5% of detail pages contain at least one massive paragraph that's impossible to scan.

## Approach

Rewrite `format_pi_text` with a three-stage pipeline:

### Stage 1: Pre-process — strip section header, normalize xrefs

1. Strip the top-level section header (`"1 INDICATIONS AND USAGE"`) — existing logic, keep it
2. Remove parenthesized cross-references from the *lead-in* paragraph (the text before any sub-section heading): `( 1.1 )` → just remove them, since they're navigation aids that duplicate the sub-section structure
3. Detect the "indicated/recommended for" lead-in pattern and split it into a lead sentence + bullet list items at the remaining `( N.N )` cross-ref boundaries

### Stage 2: Split at sub-section headings

1. Use a new split regex that requires the heading number to be at the start of a line or after a period+space (not inside parenthesized cross-refs): `r"(?<=\.\s)(\d+(?:\.\d+)+\s+[A-Z])|^(\d+(?:\.\d+)+\s+[A-Z])"`
2. The key insight: real sub-section headings appear after a period/space (end of previous paragraph) or at the start, while cross-refs appear inside parentheses like `( 1.1 )`
3. Match the heading and split into `number` + `title` pairs, where the title extends to the first period followed by a non-caps letter (sentence boundary) or to the next sub-section number

### Stage 3: Post-process — paragraph splitting

1. After all sub-section splits are done, for any remaining body text block > 2,000 chars, split at sentence boundaries (`. ` followed by uppercase letter)
2. Each split segment becomes a separate `<p>` tag

### HTML Output Changes

- Lead-in indication paragraphs: `<p class="indication-lead">DRUG is indicated for:</p>` + `<ul class="pi-list"><li>moderate to severe plaque psoriasis ...</li><li>active psoriatic arthritis ...</li></ul>`
- Sub-section headings: unchanged — `<h4 class="pi-subsection"><span class="sub-num">1.1</span> Plaque Psoriasis</h4>`
- Body text: stays as `<p>` blocks, but split into manageable lengths
- Cross-references: keep `<span class="xref">` styling for standalone `(5.1)` patterns that aren't part of a lead-in list

## Tasks

### Task 1: Rewrite format_pi_text with 3-stage pipeline

**Files:** build.py

**What:**
1. Replace the current `format_pi_text` function (lines 41-166) with the new 3-stage implementation
2. Stage 1 (pre-process):
   - Strip top-level section header (existing logic)
   - Detect indication lead-in pattern: `(?:indicated|recommended)\s+(?:for|as)\s+`
   - If found, extract the lead sentence up to the colon or first `( N.N )`, then split remaining text at `( N.N )` boundaries into list items
   - Remove `( N.N )` xrefs from lead-in list items (they duplicate the sub-section structure)
3. Stage 2 (sub-section split):
   - Use regex `r"(?m)^(\d+(?:\.\d+)+)\s+([A-Z][A-Za-z\s&,-]+?)(?=\s+[A-Z][A-Za-z]|$|\s+\d+\.\d)"` to match headings at line start only (after stripping section header, sub-sections should be at the start of remaining text segments)
   - Alternatively, split text at `\d+\.\d+\s+[A-Z][a-z]+` to avoid matching single-letter artifacts from `( N.N )` patterns — key requirement: title must start with a lowercase letter after the initial capital (e.g., "Plaque" not "P")
4. Stage 3 (paragraph split):
   - After heading + body assembly, for body blocks > 2,000 chars, split at `\. (?=[A-Z0-9])` boundaries
5. Keep all existing features: table/list protection, cross-ref styling, bullet list detection
